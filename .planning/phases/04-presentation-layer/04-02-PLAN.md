<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| user keyboard → event loop | crossterm key/resize events drive scroll state (presentation-only). |
| process → terminal vs pipe | render() chooses alt-screen TUI (TTY) or plain text (pipe) before any ratatui call. |
| panic → terminal state | A panic mid-render must not leave the terminal in raw/alt-screen mode. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-04 | Denial of availability | panic leaves terminal raw/alt-screen (DoS-ish UX) | mitigate | Use `ratatui::run`/`init` built-in panic-restore hook (Task 2, D-02); no other panic hook is installed so the ordering note is satisfied. NO hand-rolled lifecycle. |
| T-04-05 | Tampering | alt-screen escape codes leak into a pipe | mitigate | Branch on `std::io::stdout().is_terminal()` BEFORE any ratatui call (Task 3, Pitfall 5); non-TTY path uses plain::render (no escapes). |
| T-04-06 | Information disclosure / availability | io::Error from rendering bypasses error taxonomy | mitigate | Map render io::Error into `IntakeError` in app::run (Task 3, V7 error handling); never panic past the restore hook. No secrets to leak. |
| T-04-07 | Denial of availability | scroll offset out of range after resize → blank screen | mitigate | Re-clamp `offset = offset.min(max_scroll(..))` inside `draw` every frame (Task 2, Pitfall 4). |
</threat_model>

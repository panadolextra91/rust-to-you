# Architecture Research

**Domain:** Repository investigation CLI/TUI
**Researched:** 2026-06-02
**Confidence:** MEDIUM

## Standard Architecture

### System Overview

```text
┌─────────────────────────────────────────────────────────────┐
│                    Input / Presentation                      │
├─────────────────────────────────────────────────────────────┤
│  CLI Parser  │  App Orchestrator  │  Ratatui Report View    │
├─────────────────────────────────────────────────────────────┤
│                    Collection Layer                          │
├─────────────────────────────────────────────────────────────┤
│ GitHub Client │ Git Clone/History │ Filesystem/Infra Scanner │
├─────────────────────────────────────────────────────────────┤
│                    Analysis Layer                            │
├─────────────────────────────────────────────────────────────┤
│ Summary Stats │ Commit/Branch │ Relics │ Language │ Vibes    │
├─────────────────────────────────────────────────────────────┤
│                    Data Model Layer                          │
│ Raw snapshot → normalized snapshot → report view model      │
└─────────────────────────────────────────────────────────────┘
```

### Component Responsibilities

| Component | Responsibility | Typical Implementation |
|-----------|----------------|------------------------|
| CLI/parser | Validate repo URL and runtime options | `clap` structs and a small command entrypoint |
| Collection services | Fetch remote metadata and local repo facts | `reqwest` + `git2` + file scanning helpers |
| Analyzers | Turn raw snapshots into report-ready metrics | Pure functions over normalized data structs |
| TUI renderer | Present the report as one readable scroll | `ratatui` widgets backed by a single report model |

## Recommended Project Structure

```text
src/
├── main.rs                 # CLI entrypoint
├── app/                    # Orchestration and session lifecycle
├── cli/                    # clap models and input validation
├── github/                 # REST client and response models
├── repo/                   # clone workspace and git2 helpers
├── scan/                   # filesystem, language, and infra discovery
├── analyze/                # section analyzers and finding heuristics
├── report/                 # report view model and section formatting
└── tui/                    # ratatui widgets, scrolling, and keyboard input
```

### Structure Rationale

- **Collection is separated from analysis:** fetch once, reuse everywhere.
- **A report view model sits between analyzers and TUI:** keeps terminal rendering dumb and testable.
- **GitHub and local repo logic stay separate:** prevents remote API assumptions from leaking into archaeology code.

## Architectural Patterns

### Pattern 1: Investigation Pipeline

**What:** Run the app as a staged pipeline: intake → collect → analyze → render.
**When to use:** Always for the primary command.
**Trade-offs:** Easy to reason about, but requires careful snapshot design up front.

**Example:**
```rust
// Sequential pipeline — no async runtime in V1.
let snapshot = collector.collect(repo_url)?;
let report = analyzer.build_report(&snapshot)?;
tui::render(report)?;
```

### Pattern 2: Normalized Snapshot Model

**What:** Convert API payloads, git history, and filesystem findings into one internal snapshot type.
**When to use:** Before any section-specific analyzer runs.
**Trade-offs:** Slight upfront modeling cost, but dramatically cleaner analyzers.

**Example:**
```rust
struct InvestigationSnapshot {
    repo: RepoMetadata,
    history: HistoryFacts,
    filesystem: FilesystemFacts,
    branches: BranchFacts,
}
```

### Pattern 3: Pure Section Analyzers

**What:** Each report section is generated by a pure analyzer over snapshot data.
**When to use:** For metrics, heuristics, and interesting findings.
**Trade-offs:** More modules, but much better testability and clearer ownership.

## Data Flow

### Request Flow

```text
Repo URL
    ↓
CLI validation
    ↓
Minimal API metadata + full clone + local scan
    ↓
Normalized investigation snapshot
    ↓
Section analyzers + findings engine
    ↓
Report view model
    ↓
Scrollable Ratatui report
```

### State Management

```text
InvestigationSession
    ↓ owns
InvestigationSnapshot
    ↓ feeds
ReportModel
    ↓ displayed by
TuiState { scroll_offset, viewport_height }
```

### Key Data Flows

1. **Overview flow:** GitHub metadata → summary section cards and header information.
2. **Archaeology flow:** local git history + filesystem scan → commit, branch, relic, and language sections.
3. **Narrative flow:** analyzer outputs → vibes, findings, and verdict sections.

## Scaling Considerations

| Scale | Architecture Adjustments |
|-------|--------------------------|
| Small/typical OSS repos | Single-process pipeline is enough |
| Very branchy or long-lived repos | Keep branch/ref enumeration local, and cache intermediate facts from expensive history passes |
| Huge monorepos | Keep the clone full but bound history walks; avoid loading full trees into memory at once |

### Scaling Priorities

1. **First bottleneck:** repository collection time — fix with a single metadata call, full clone, and bounded/selective history walks.
2. **Second bottleneck:** report generation over giant histories — fix with bounded analysis passes and cached intermediate facts.

## Anti-Patterns

### Anti-Pattern 1: Mixing rendering with analysis

**What people do:** Compute metrics inside widget code.
**Why it's wrong:** Makes sections hard to test and impossible to reuse for future JSON export.
**Do this instead:** Build a report model first, render second.

### Anti-Pattern 2: Treating GitHub UI HTML as the data source

**What people do:** Scrape pages because "the data is already there."
**Why it's wrong:** Fragile, harder to rate-limit correctly, and semantically noisy.
**Do this instead:** Use GitHub APIs for remote metadata and `git2` for local archaeology.

## Integration Points

### External Services

| Service | Integration Pattern | Notes |
|---------|---------------------|-------|
| GitHub REST API | Single authless GET `/repos/{owner}/{repo}` | Only for stars/forks/description; everything else comes from git2. Surface clear fallback errors. |
| Remote git host | Read-only full clone | Full clone for complete history; bound expensive analysis passes rather than limiting clone depth. |

### Internal Boundaries

| Boundary | Communication | Notes |
|----------|---------------|-------|
| `github` ↔ `analyze` | typed models | Keep raw HTTP response shapes out of analyzer logic. |
| `repo`/`scan` ↔ `analyze` | normalized facts | History and filesystem data should be pre-aggregated before scoring. |
| `report` ↔ `tui` | view-model structs | TUI should not need to know how metrics were computed. |
